TLS works over TCP.

Then there is the initialization of the TLS handshake. After installing TCP, the client sends the specification to the server in plain text. The server approves the version of the protocol used, selects the encryption method from the provided list, attaches its certificate, and sends a response to the client.
The client verifies the sent certificate and initiates either RSA or Diffie-Hellman key exchange.

The server processes the message sent by the client, verifies the MAC, and sends the Finished message to the client in encrypted form.
The client decrypts the received message, checks the MAC, and if everything is fine, the connection is considered established and the application data exchange begins.

  1. What version of TLS is used? Describe which versions are not safe to use and under which conditions.
  2. What ciphersuite is used? Make a small list of ciphersuite that you would recommend to use and why.

Here I use TLSv1.3. This is the best solution at the moment.

SSL2, SSL3, Tlsv1.1 contain serious vulnerabilities, so it is strongly recommended never to use them.

TLSv1.2 also contains vulnerabilities (vulnerable ciphers), but it can be used for older systems that do not support TLSv1.3.

In my case, the cipher suite is TLS_CHACHA20_POLY1305_SHA256.

TLS - Protocol.
ChaCha stream cipher and Poly1305 authenticator (CHACHA20_POLY1305) - Encryption.
SHA256 - Hash.

Let’s start with TLSv1.3. So, the five ciphersuites that TLS 1.3 supports:

  1. TLS_AES_128_GCM_SHA256;
  2. TLS_AES_256_GCM_SHA384;
  3. TLS_CHACHA20_POLY1305_SHA256;
  4. TLS_AES_128_CCM_SHA256;
  5. TLS_AES_128_CCM_8_SHA256.

All of them are AEAD (authenticated encryption with attached data). This means that they are doing several things at once at the same time:

  • Encrypt data with a key. This preserves the confidentiality of the data;
  • Authenticate data using a key. This preserves integrity;
  • Authorize (again, using a key) unencrypted data. This gives the same guarantees in terms of integrity and the ability to create valid messages for any data that needs to be validated.

The CCM mode has flaws in its design. Therefore, I cannot recommend the 4th and 5th options.

Among these ciphersuites best solutions are TLS_AES_256_GSM_SHA384 and TLS_CHACHA20_POLY1305_SHA256 in order of speed preference.

There are many ciphers in TLSv1.2. But I would like to recommend the following options:

  1. TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384;
  2. TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256;
  3. TLS_DHE_RSA_WITH_AES_256_GCM_SHA384.

a. When do you think TLS proxy is used for security purposes in the industry?

TLS provides encryption and authenticity of communication over the Internet.
A connection is being intercepted by a TLS proxy when it inspects incoming traffic to block malicious connections.
A high-speed redundant network is used to run the TLS proxy to protect against DDoS attacks.
It also provides a high level of anonymity (the network behind a proxy is hidden), while maintaining the convenience and simplicity for users inherent in proxy servers.

b. What are techniques that are used to protect applications (desktop, web, mobile, etc.) from TLS proxying?

To detect a proxy server, we have to get the certificate that the client actually receives. Such as a web browser, and then compare it with the valid certificate presented by the server that the client is contacting. The discrepancy indicates that some proxy server, benevolent or malicious, intercepts client traffic to this particular server.

The client only trusts the root certificate authority stored locally, and any certificate/CA must be issued directly or indirectly by that CA. This proxy certificate authority must be explicitly installed in the browser/OS.

Thus while you cannot make a proxy to not intercept and modify the TLS connection this modification will be detected immediately and the connection will be stopped before any data are transferred. Only if you explicitly trust the proxy by importing the proxy CA the TLS interception is possible.

In mobile devices, a mechanism called SSL pinning is used for this purpose.
The browser also uses a locally stored certificate

Configuring openssl.cnf for the Root CA Certificate

To create my certificates, I created an openssl.cnf configuration file, which I copied from /etc/ssl. So let’s see what I have changed in it.

The [ CA_default ] block is the name of the section containing the default CA configuration. There I changed the following lines:

dir = /home/st12/Desktop/gotls
certificate = $dir/certs/ca_cert.pem
private_key = $dir/private/ca_key.pem

Only the dir option was important here, the other two are optional.

The values under the [ req ] section are applied when creating Certificate Signing Requests (CSR) or Certificates. The x509_extensions key specifies the name of a section that contains the extensions that we want to be included in the certificate.

[ req ]
default_bits            = 4096
default_md              = sha256
default_keyfile         = private_key.pem
distinguished_name      = req_distinguished_name
x509_extensions         = st12_ca
string_mask             = nombstr

The [ req_distinguished_name ] key defines how OpenSSL obtains the information needed to fill in the distinctive name of the certificate. I have given a few default values to simplify the process.

[ req_distinguished_name ]
countryName_default             = RU
stateOrProvinceName_default     = Tatarstan
localityName_default            = Innopolis
0.organizationName_default      = Innopolis University

Then I configured the extensions that I would use with the openssl certificate chain. I will use the st12_ca extension to create the root CA certificate and the st12_intermediate_ca extension for the intermediate CA certificate.

[ st12_ca ]
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid:always,issuer
basicConstraints = critical,CA:true

[ st12_intermediate_ca ]
subjectKeyIdentifier = hash
authorityKeyIdentifier = keyid:always,issuer
basicConstraints = critical, CA:true, pathlen:1
keyUsage = critical, digitalSignature, cRLSign, keyCertSign

To create a root CA certificate (ca_cert.pem), I need to use the root CA key (ca_key.pem).

Generating an Intermediate CA key and CSR

Generate intermediate CA key (ca-intermediate.key) using openssl genrsa:

After that, I used this key to create a certificate signing request (CSR) as shown in Figure 24:

Certificate Chain

As I said earlier, I’m going to use mitmproxy to create a server certificate. This solution will allow me to create a chain for any server.
To do this, I need to run the following commands:

//Creating a new directory for mitmproxy
mkdir mitmproxy
cd mitmproxy/
//Copying the intermediate key and certificate
cp ../intermediate/certs/intermediate_ca_cert.pem mitmproxy-ca-cert.pem
cp ../intermediate/private/intermediate_ca_key.pem mitmproxy-ca-key.pem
//Creating the necessary files
touch mitmproxy-ca.pem
cat mitmproxy-ca-key.pem >> mitmproxy-ca.pem
cat mitmproxy-ca-cert.pem >> mitmproxy-ca.pem
// Creating symbolic link
ln -s mitmproxy-ca-cert.pem mitmproxy-ca-cert.cer
// Creating PKCS#12 file
openssl pkcs12 -export -out mitmproxy-ca-cert.p12 -inkey mitmproxy-ca-key.pem -in mitmproxy-ca-cert.pem -password pass:k1ll
// Creating an archive
tar czvf ../mitmproxy.tgz *

As a result, I got an archive containing all the necessary files. Now let’s put them in mitmproxy.

mkdir -p ~/.mitmproxy
tar xzf ../mitmproxy.tgz -C ~/.mitmproxy/
